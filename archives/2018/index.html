Spring Cloud Finchley 在业务系统中的实践总结

前言

　　谈及Spring Cloud，就不可避免的要先讲讲近几年一直很火热的微服务概念。随着公司业务的发展，每一个技术项目都会经历从小到大、从简至繁，再到化繁为简的过程，那么如何合理的化繁为简，是一个亘古不变的值得持续讨论的话题。
　　如果将微服务理解为是可以独立部署、水平扩展、独立访问的服务单元，Spring Cloud就是这些微服务的大管家。微服务为什么需要大管家？
　　
　　

　　
　　上图虽然夸张，但确是我司业务在微服务进程中所遇到的其中一个问题。宜人贷的业务系统从2012年上线以来，历经无数次的迭代与服务拆分以后，现如今已裂变成了数百个子项目。虽然对于某一个微服务来说，维护成本大大降低，但业务是整体的，如果没有所谓的大管家，那么维护微服务集群一定是一个灾难。

简谈我们微服务化的历史进程

　　在最初的微服务化进程中，也就是从2013年开始，我们引入了当时理念较为先进的Dubbo作为服务化治理的核心框架一直沿用至今。那么，为什么我们还要使用Spring Cloud？
　　首先，Dubbo自阿里巴巴开源后的很长一段时间长期停止了维护（2017年7月31日阿里团队又宣布重点维护），而Spring Cloud确是一直迭代至今，在很多技术理念上Dubbo相比Spring Cloud而言已不具优势，Dubbo所具备的功能也依然成为Spring Cloud的一个子集。
　　其次，Spring Cloud基于Spring Boot发展而来，其提供了一整套满足微服务的、可插拔式的一站式解决方案，简单易用。
　　最后，在我司业务中的App与App Server端，一直是Dubbo大管家所失控的法外之地，仅几个业务线，所部署的公网域名就多达几十个，可见这里服务之间繁杂的关系的维护成本，这部分项目已然是上图中的一个缩影。目前Spring Cloud在整个社区有着非常高的活跃度，那么选择Spring Cloud来治理当然是不二之选。

基于Spring Cloud我们能做哪些事？

　　引用官方说明描述一下Spring Cloud的核心特性以及支持其特性的核心组件。

Features　　

　　Spring Cloud focuses on providing good out of box experience for typical use cases and extensibility mechanism to cover others.

  • Distributed/versioned configuration
  • Service registration and discovery
  • Routing
  • Service-to-service calls
  • Load balancing
  • Circuit Breakers
  • Global locks
  • Leadership election and cluster state
  • Distributed messaging

Main Projects

  • Spring Cloud Config
  • Spring Cloud Netflix(Eureka, Hystrix, Zuul, Archaius, etc.).
  • Spring Cloud Bus
  • Spring Cloud for Cloud Foundry
  • Spring Cloud Open Service Broker
  • Spring Cloud Cluster
  • Spring Cloud Consul
  • Spring Cloud Security
  • Spring Cloud Sleuth
  • Spring Cloud Data Flow
  • Spring Cloud Stream
  • Spring Cloud Task
  • Spring Cloud Zookeeper
  • Spring Cloud for Amazon Web Services
  • Spring Cloud Connectors
  • Spring Cloud Starters
  • Spring Cloud CLI
  • Spring Cloud Contract
  • Spring Cloud Gateway
  • Spring Cloud OpenFeign
  • Spring Cloud Pipelines
  • Spring Cloud Function

　　可以看到，Spring Cloud提供了非常多的核心组件，这些组件已经将微服务的所有需求都涵盖了。
